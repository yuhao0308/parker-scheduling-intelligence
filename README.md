# United Hebrew Scheduling Intelligence

**Version:** `0.1.0` &nbsp;•&nbsp; **Status:** Proof of Concept &nbsp;•&nbsp; **Repo:** `parker-scheduling-intelligence`

A decision-support system for skilled-nursing facilities. It generates monthly staff schedules and ranks replacement candidates for same-day call-outs with plain-English rationale generated by a local LLM.

The tool sits on top of the facility's existing workforce-management system (e.g. SmartLinx, UKG/Kronos) — it is **read-only system of record**, never a payroll writer.

## What it does

Two coordinator workflows:

| Workflow | What the system does |
|---|---|
| **Monthly schedule generation** | Builds a full month of shift assignments respecting licenses, full/part-time hour caps, clinical experience, rest windows, and unit exclusions. Surfaces a scenario badge (Ideal / Moderate / Critical) and per-slot warnings. |
| **Same-day call-out replacement** | When a staff member calls out, returns a ranked list of replacement candidates (default top-10) with a Claude-generated rationale per candidate. The coordinator selects with one click; non-top picks prompt for a brief reason for compliance tracking. |

Candidates are ranked by a deterministic weighted score combining overtime headroom, clinical fit, float penalty, proximity, seniority, equity, and historical willingness. Weights live in [`config/scoring_weights.yaml`](config/scoring_weights.yaml) and are hot-tunable.

A more detailed product description lives in [BUSINESS_OVERVIEW.md](BUSINESS_OVERVIEW.md).

## Architecture

```
┌────────────────────────┐        ┌──────────────────────────┐
│  Next.js frontend      │  HTTP  │  FastAPI backend         │
│  (App Router, React 19)│ ─────► │  app/main.py             │
└────────────────────────┘        │  ├─ routes/              │
                                  │  ├─ services/            │
                                  │  │   ├─ scheduler        │
                                  │  │   ├─ recommendation   │
                                  │  │   ├─ scoring          │
                                  │  │   ├─ overtime         │
                                  │  │   ├─ rationale (LLM)  │
                                  │  │   └─ workload         │
                                  │  └─ integrations/        │
                                  │       └─ timeclock/      │
                                  └────────┬─────────────────┘
                                           │
                          ┌────────────────┼────────────────┐
                          ▼                ▼                ▼
                  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
                  │ PostgreSQL16 │ │ Ollama (LLM) │ │ Timeclock CSV│
                  │ (asyncpg)    │ │ qwen3:8b     │ │ (Kronos export)│
                  └──────────────┘ └──────────────┘ └──────────────┘
```

Key services:

- **`scheduler`** — month-level assignment engine.
- **`recommendation`** — call-out replacement pipeline (filter → score → rationale).
- **`overtime`** — RN biweekly + non-RN weekly overtime tracking; enforces voluntary-OT-only when configured.
- **`workload`** — per-employee hours ledger derived from schedule + ingested punches.
- **`rationale`** — generates plain-English candidate explanations via local Ollama.
- **`integrations/timeclock`** — ingests Kronos-format punch data into daily/pay-period rollups.

## Getting started

### Prerequisites

- Python **3.12+**
- Node.js **20+** (for the frontend)
- Docker + Docker Compose (recommended for local Postgres)
- [Ollama](https://ollama.com) running locally with `qwen3:8b` pulled, if you want LLM rationale generation
  ```bash
  ollama pull qwen3:8b
  ```

### 1. Clone & configure

```bash
git clone git@github.com:yuhao0308/parker-scheduling-intelligence.git
cd parker-scheduling-intelligence
cp .env.example .env   # adjust values as needed
```

### 2. Bring up Postgres

```bash
docker compose up -d db
```

### 3. Install backend deps & run migrations

```bash
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"
alembic upgrade head
```

### 4. Seed demo data (optional)

```bash
python scripts/seed_dev_data.py
python scripts/ingest_timeclock.py    # loads sample timeclock punches
```

### 5. Run the backend

```bash
uvicorn app.main:app --reload
```

API is now live at <http://localhost:8000> (OpenAPI docs at `/docs`).

### 6. Run the frontend

```bash
cd frontend
npm install
npm run dev
```

Open <http://localhost:3000>.

### Run everything via Docker

```bash
docker compose up --build
```

## Configuration

All runtime configuration is environment-driven (Pydantic Settings, see [`app/config.py`](app/config.py)). Copy `.env.example` to `.env` and tune.

| Variable | Default | Purpose |
|---|---|---|
| `DATABASE_URL` | `postgresql+asyncpg://united_hebrew:united_hebrew@localhost:5432/united_hebrew_scheduling` | Postgres connection (`postgres://` and `postgresql://` are auto-rewritten to `+asyncpg`). |
| `OLLAMA_BASE_URL` | `http://localhost:11434` | Ollama HTTP endpoint. |
| `OLLAMA_MODEL` | `qwen3:8b` | Model used for candidate rationale. |
| `FACILITY_ZIP_CODE` | `11375` | Anchor for proximity scoring. |
| `LOG_LEVEL` | `INFO` | structlog level. |
| `SHADOW_MODE` | `false` | When `true`, recommendations are computed but not surfaced — for safe pre-launch validation. |
| `VOLUNTARY_OT_ONLY` | `true` | When `true`, candidates whose acceptance would trigger overtime are eliminated outright (NY Labor Law §167 compliance). |
| `DEMO_MODE` | `true` | Compresses outreach/confirmation timers to 15s for live demos; UI labels remain truthful (`"15 minutes"`, `"2 hours"`). |
| `UKG_*` | empty | Optional UKG/Kronos write-back credentials (disabled by default). |

Scoring weights — [`config/scoring_weights.yaml`](config/scoring_weights.yaml):

| Weight | Default | Notes |
|---|---|---|
| `overtime_headroom` | 0.45 | Largest financial lever. |
| `clinical_fit` | 0.22 | Subacute vs long-term match. |
| `float_penalty` | 0.13 | Protects home-unit assignment + new hires. |
| `seniority` | 0.08 | Tenure / experience. |
| `equity` | 0.05 | Surfaces under-used per-diem staff. |
| `proximity` | 0.05 | Tiebreaker only. |
| `willingness` | 0.02 | Historical acceptance. |

## Operational notes

- **Data sync** — Staff roster, schedule, hours, callouts, and units are pulled from the facility's WFM system; the tool never writes back unless `UKG_WRITE_BACK_ENABLED=true`.
- **LLM warm-up** — `app.main.lifespan` fires a background `warm_ollama` task so the first call-out doesn't pay the cold-load cost.
- **Operational day boundary** — Shift accounting uses an 11 PM rollover to match clinical scheduling conventions.
- **RN dual-track overtime** — RNs use biweekly OT accounting (`rn_shifts_remaining_amber`); other roles use weekly hours (`standard_hours_remaining_amber`).
- **Health probes** — `GET /health` and `GET /health/ollama` check the API and local LLM reachability respectively.
