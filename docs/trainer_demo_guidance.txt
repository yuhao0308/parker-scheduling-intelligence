9. MONTHLY AUTO-SCHEDULING: BACKEND LOGIC
==================================================

If asked to explain the backend logic in detail, use this structure.

9.1 Inputs used by the monthly scheduler

The scheduler uses:

- active units
- active staff
- each employee's license type
- employment class such as full-time, part-time, or per diem
- home unit and unit typology
- cross-training information
- hire date
- baseline hours ledger
- configurable scoring weights
- the requested month and year

For demo scenarios, it can also artificially reduce the available staff pool to simulate moderate or critical shortage conditions.

9.2 How the monthly schedule is generated

The scheduler:

- clears the existing schedule entries for that month
- loads all active units
- sorts units so Subacute units are considered before Long-Term units
- loads the available staff pool
- walks through every day of the month
- for each day, walks through each unit
- for each unit, walks through each shift in order: Day, Evening, Night

9.3 Staffing preference by unit type

In the current demo, the simplified staffing model uses one staff member per unit per shift.

The unit preference is:

- Subacute: prefer licensed staff first, then certified staff as fallback
- Long-Term: prefer certified staff first, then licensed staff as fallback

This is important because it shows that the system is not filling slots randomly. It is trying to preserve clinically appropriate coverage by unit type.

9.4 Hard eligibility checks before assignment

Before assigning someone to a shift, the scheduler filters candidates to remove people who should not be considered for that slot.

The current monthly scheduler enforces:

- license bucket matching
- no duplicate assignment to the exact same slot
- rest-window protection through operational-day shift counting
- weekly scheduling caps by employment class

The weekly caps in the current demo are:

- full-time: up to 41.25 hours
- part-time: up to 24.75 hours
- per diem: up to 41.25 hours

The rest-window logic is based on Parker's operational day, which runs from 11:00 PM to 11:00 PM, not midnight to midnight.

That matters especially for Night shift, because a Night shift starting at 11:00 PM is treated as part of the next operational day.

9.5 How the monthly scheduler chooses the best person

Once ineligible people are filtered out, the remaining candidates are scored.

For monthly scheduling, the key scoring dimensions are:

- overtime headroom
- clinical fit
- float penalty

Proximity is intentionally neutralized in monthly scheduling because travel distance is much more relevant for emergency call-outs than for planned monthly scheduling.

The scheduler then assigns the highest-scoring surviving candidate.

9.6 What happens if nobody fits

If no candidate passes the filters and weekly caps, that slot is left unfilled.

The system records a warning and the UI surfaces the gap.

This is a strength, not a weakness, because the system is showing the real staffing constraint instead of hiding it with an unsafe assignment.

9.7 Scenario labels

After generation, the scheduler classifies the month as:

- ideal: no unfilled slots
- moderate: up to 10 percent unfilled
- critical: more than 10 percent unfilled

This gives the user an immediate sense of staffing severity.

9.8 Safe summary line

"So the monthly scheduler is not just filling slots. It is applying staffing constraints, unit priorities, overtime awareness, and clinical fit rules to build the best possible month from the current database state."

9.9 Monthly scheduler mathematical demonstration

If your trainer asks for the actual calculation, use this exact structure.

Monthly scheduler decision pipeline for one slot:

Step 1. Choose preferred bucket by unit type

- Subacute slot: licensed first, certified second
- Long-Term slot: certified first, licensed second

Step 2. Filter out ineligible staff

The candidate must satisfy all of the following:

- correct license bucket
- not already scheduled in that exact slot
- does not violate the rest-window rule
- does not exceed weekly assignment cap after adding this shift

Weekly cap math:

- shift_duration = 8.25 hours
- full_time_cap = 41.25
- part_time_cap = 24.75
- per_diem_cap = 41.25

The cap test is:

current_weekly_hours + 8.25 <= weekly_cap

If false, the candidate is removed.

Step 3. Compute the score for each surviving candidate

The monthly scheduler uses the same weighted score function as the recommendation engine, but with proximity fixed to 0.5 for everyone, and seniority, equity, and willingness left at 0 in the actual scheduler call.

Monthly scheduler score formula:

MonthlyScore
= 0.45 * OvertimeHeadroom
+ 0.22 * ClinicalFit
- 0.13 * FloatPenalty
+ 0.05 * 0.5

Since 0.05 * 0.5 = 0.025, this becomes:

MonthlyScore
= 0.45 * OvertimeHeadroom
+ 0.22 * ClinicalFit
- 0.13 * FloatPenalty
+ 0.025

That last 0.025 is a constant added to every surviving candidate, so in practice it does not change the ranking order. It just means the ranking is effectively driven by:

- overtime headroom
- clinical fit
- float penalty

Step 4. Assign the highest-scoring candidate

The selected person is:

argmax(MonthlyScore)

If no one survives the filter and cap checks, the slot stays unfilled.

9.10 Monthly scheduler scoring components, mathematically

A. Overtime headroom for CNA, PCT, and LPN

Definitions:

- weekly_overtime_threshold = 37.5
- soft_cap = 62.5
- shift_duration = 8.25
- hours_after_shift = hours_this_cycle + 8.25

Overtime trigger flag:

would_trigger_ot = (hours_after_shift > 37.5)

Normalized overtime headroom:

If hours_this_cycle <= 37.5:
OvertimeHeadroom = 1.0

If 37.5 < hours_this_cycle < 62.5:
OvertimeHeadroom = 1 - ((hours_this_cycle - 37.5) / 25.0)

If hours_this_cycle >= 62.5:
OvertimeHeadroom = 0.0

Interpretation:

- 1.0 means strong straight-time headroom
- 0.0 means the person is already at or beyond the soft cap

B. Overtime headroom for RN

RN overtime uses two tracks:

1. Daily overtime
2. Biweekly shift-count overtime

Daily overtime rule:

- if the RN already has a shift in the same operational day, adding another shift makes daily_headroom = 0
- otherwise daily_headroom = 1

Biweekly overtime rule:

- threshold = 10 shifts in the biweekly cycle
- the 11th shift triggers overtime

Formula:

biweekly_headroom = max(0, (10 - shift_count_this_biweek) / 10)

Final RN overtime headroom:

OvertimeHeadroom = min(daily_headroom, biweekly_headroom)

C. Clinical fit values

ClinicalFit = 1.0 if any of the following is true:

- home unit matches target unit
- candidate is cross-trained for the target unit
- candidate home typology matches target typology

ClinicalFit = 0.8 if:

- candidate is Subacute and target is Long-Term

ClinicalFit = 0.0 if:

- candidate is Long-Term only and target is Subacute

D. Float penalty values

FloatPenalty = 0.0 if:

- target unit is the candidate's home unit

FloatPenalty = 0.3 if:

- different unit, same typology

FloatPenalty = 0.6 if:

- different unit, different typology

If tenure < 6 months:

FloatPenalty = min(base_penalty * 1.5, 1.0)

9.11 Monthly scheduler worked example

Suppose we are filling a Subacute Day shift and have two surviving candidates.

Candidate A:

- CNA or licensed-bucket equivalent appropriate for the slot
- hours_this_cycle = 30.0
- home unit = target unit
- same unit fit
- no float

Candidate B:

- eligible for the slot
- hours_this_cycle = 45.0
- from Long-Term, not home unit
- floating cross-typology
- not a new hire

Candidate A calculations:

OvertimeHeadroom = 1.0
ClinicalFit = 1.0
FloatPenalty = 0.0

MonthlyScore
= 0.45(1.0) + 0.22(1.0) - 0.13(0.0) + 0.025
= 0.45 + 0.22 + 0 + 0.025
= 0.695

Displayed as an approximate score concept:
0.695 x 100 = 69.5

Candidate B calculations:

OvertimeHeadroom = 1 - ((45.0 - 37.5) / 25.0)
= 1 - (7.5 / 25.0)
= 0.7

ClinicalFit = 0.0
FloatPenalty = 0.6

MonthlyScore
= 0.45(0.7) + 0.22(0.0) - 0.13(0.6) + 0.025
= 0.315 + 0 - 0.078 + 0.025
= 0.262

Candidate A wins because:

- much better overtime headroom
- perfect unit fit
- no float penalty

This is the kind of mathematical demonstration that is strong in a trainer conversation, because it shows exactly why one person outranks another.

13. RECOMMENDATION MODEL: BACKEND LOGIC
==================================================

13.1 Inputs used by the recommendation engine

The recommendation flow uses:

- the employee who called out
- the target unit
- the shift date
- the shift label
- the active staff roster
- home unit information
- unit typology
- cross-training data
- current schedule state
- PTO records
- unit exclusion records
- hours ledger
- recent shift history
- configurable scoring weights

13.2 Step 1: infer the required license bucket

The system first looks at the employee who called out and infers what license bucket the replacement must match.

In practice:

- if the call-out was from a licensed role, the replacement must come from the licensed bucket
- if the call-out was from a certified role, the replacement must come from the certified bucket

This prevents unsafe substitutions.

13.3 Step 2: build the current schedule context

The system builds context around the target date and shift, including:

- who is already scheduled
- who is on PTO
- which shifts each candidate recently worked
- current per-unit coverage counts

This allows it to reason from the actual live scheduling state instead of from a disconnected ranking table.

13.4 Step 3: hard filters

Before scoring, the engine removes anyone who fails a hard rule.

The main hard filters are:

- license mismatch
- already scheduled for that shift
- on PTO
- excluded from that unit
- rest-window violation
- source-unit coverage floor protection

The source-unit coverage floor is important. It prevents the system from fixing one problem by creating a new staffing problem in the person's home unit.

13.5 Step 4: scoring the valid candidates

After filtering, the remaining candidates are scored using weighted dimensions.

Current configured weights:

- overtime headroom: 45 percent
- clinical fit: 22 percent
- float penalty: 13 percent
- seniority: 8 percent
- proximity: 5 percent
- equity: 5 percent
- willingness: 2 percent

The important message is:

"Overtime is weighted highest because that is the strongest financial and operational pressure Parker identified. Clinical fit is next because we still need the right people in the right places. The rest of the signals help break ties and improve quality."

13.6 What each scoring dimension means

Overtime headroom:
- prefers people with more straight-time capacity left
- helps avoid avoidable overtime

Clinical fit:
- rewards home-unit match
- rewards cross-training for the target unit
- rewards same typology match
- treats Subacute to Long-Term as acceptable
- strongly penalizes Long-Term-only to Subacute

Float penalty:
- penalizes moving people away from their home unit
- penalizes cross-typology floating more heavily
- penalizes newer staff more heavily when floating

Seniority:
- gives modest credit for longer tenure
- saturates at 10 years in the current configuration

Proximity:
- estimates travel distance from the employee zip code to the facility
- used as a light signal, mainly for tie-breaking

Equity:
- helps surface under-used per-diem staff who have been dormant
- this is meant to avoid overusing the same people every time

Willingness:
- currently a very low-weight signal
- right now it defaults to a neutral baseline until more override history is available

13.7 Overtime logic details

For CNA, PCT, and LPN:

- weekly overtime threshold is 37.5 hours
- the model uses a soft-cap logic up to 62.5 hours
- beyond that point, the headroom signal drops to zero

For RN:

- daily overtime can be triggered by a second shift in the same operational day
- biweekly overtime can be triggered on the 11th shift in a 14-day cycle

This matters because different roles may have different overtime behavior, and the engine takes that into account before ranking candidates.

13.8 Step 5: rank and explain

The engine sorts candidates by score and returns the top results.

The UI shows:

- rank
- name
- license
- employment class
- home unit
- score
- score breakdown
- rationale

The rationale explains the recommendation in plain language, such as:

- hours remaining before overtime
- experience or unit fit
- whether this is the home unit or a float
- approximate travel distance

13.9 Step 6: human-in-the-loop selection

The coordinator makes the final decision.

If the user selects the number one candidate, the system logs that selection directly.

If the user selects someone other than number one, the system asks for a reason and logs the override.

This is an important governance and compliance feature because it preserves human judgment while still keeping a traceable decision record.

13.10 Safe summary line

"So the recommendation engine first eliminates unsafe options, then ranks the valid ones by Parker's operational priorities, and finally explains the result so the coordinator can make the final call."

13.11 Recommendation engine exact score formula

For each surviving candidate, the recommendation engine computes:

RecommendationScore
= 0.45 * OvertimeHeadroom
+ 0.22 * ClinicalFit
+ 0.05 * Proximity
+ 0.08 * Seniority
+ 0.05 * Equity
+ 0.02 * Willingness
- 0.13 * FloatPenalty

The UI then displays:

DisplayedScore = round(RecommendationScore * 100)

So a raw score of 0.789 is shown as about 79 out of 100.

13.12 Recommendation engine full calculation process

Step 1. Start with all active staff.

Step 2. Infer required replacement bucket from the person who called out:

- licensed bucket if the call-out employee was RN or LPN
- certified bucket if the call-out employee was CNA or PCT

Step 3. Apply hard filters.

A candidate is removed if any of these is true:

- wrong license bucket
- already scheduled for that shift
- on PTO
- excluded from the target unit
- violates rest-window rule
- removing them from their source unit would drop that source unit below minimum coverage

Source-unit coverage floor math:

For a candidate currently rostered on their home unit during that same shift:

- if licensed_count - 1 < minimum_licensed_for_home_unit, reject
- if certified_count - 1 < minimum_certified_for_home_unit, reject

Current minimum licensed rule in the loader:

- Subacute home unit minimum licensed = 2
- Long-Term home unit minimum licensed = 1

Current minimum certified rule in the loader:

- required_ratio if present
- otherwise default = 4

Step 4. Score every survivor using the weighted formula.

Step 5. Sort descending by score.

Step 6. Return the top N candidates with explanation.

13.13 Recommendation engine scoring components, mathematically

A. Overtime headroom

The overtime formulas are the same ones used above.

For CNA, PCT, and LPN:

If hours_this_cycle <= 37.5:
OvertimeHeadroom = 1.0

If 37.5 < hours_this_cycle < 62.5:
OvertimeHeadroom = 1 - ((hours_this_cycle - 37.5) / 25.0)

If hours_this_cycle >= 62.5:
OvertimeHeadroom = 0.0

The overtime trigger flag still uses projected post-shift hours:

would_trigger_ot = (hours_this_cycle + 8.25 > 37.5)

For RN:

daily_headroom = 0 if the RN already has a shift in the same operational day, else 1

biweekly_headroom = max(0, (10 - shift_count_this_biweek) / 10)

OvertimeHeadroom = min(daily_headroom, biweekly_headroom)

B. Proximity

Distance is estimated from candidate ZIP code to facility ZIP code.

If ZIP lookup fails, the code uses:

distance_miles = 50.0

Normalized proximity score:

Proximity = max(0, 1 - distance_miles / 30)

Examples:

- 0 miles -> 1.0
- 3 miles -> 0.9
- 9 miles -> 0.7
- 30 miles -> 0.0
- anything above 30 miles -> 0.0

C. Clinical fit

ClinicalFit = 1.0 for:

- home unit match
- exact target-unit cross-training
- same unit typology

ClinicalFit = 0.8 for:

- Subacute staff covering Long-Term

ClinicalFit = 0.0 for:

- Long-Term-only staff covering Subacute

D. Float penalty

FloatPenalty = 0.0 for home unit
FloatPenalty = 0.3 for different unit, same typology
FloatPenalty = 0.6 for different unit, different typology

If tenure < 6 months:

FloatPenalty = min(base_penalty * 1.5, 1.0)

E. Seniority

Tenure in years:

tenure_years = max(0, (reference_date - hire_date) / 365.25)

Normalized seniority score:

Seniority = min(1.0, tenure_years / 10.0)

Examples:

- 2 years -> 0.2
- 8 years -> 0.8
- 12 years -> 1.0

F. Equity

This boost applies only to per-diem staff.

If employment class is not PER_DIEM:

Equity = 0

If per diem and never worked:

Equity = 1.0

If per diem and days_since_last_shift <= 30:

Equity = 0.0

If per diem and days_since_last_shift > 30:

Equity = min(1.0, (days_since_last_shift - 30) / 30)

Examples:

- 45 idle days -> (45 - 30) / 30 = 0.5
- 60 idle days -> 1.0

G. Willingness

Current demo value:

Willingness = 0.5

This is intentionally low-weight until more acceptance history is available.

19. HIGH-PROBABILITY QUESTIONS AND BEST ANSWERS
==================================================

Q1. What problem is this demo solving?

Answer:
"It solves two related staffing problems: building monthly coverage faster and handling same-day call-outs faster. In both cases, the goal is to reduce manual searching, reduce avoidable overtime, and make decisions more consistent and explainable."


Q2. Is this really machine learning?

Answer:
"In the current demo, the safest description is AI-assisted decision support. It uses business-rule filtering, weighted scoring, and transparent rationale generation. That is deliberate, because in staffing and healthcare operations, explainability and control are very important."

Optional stronger answer:
"So rather than presenting it as a black-box predictive model, we present it as an explainable intelligence engine. That is usually more credible for this kind of use case."


Q3. Why is overtime weighted the highest?

Answer:
"Because overtime is one of the most expensive and operationally sensitive staffing pressures. If two people are both clinically acceptable, the system should prefer the one who helps us avoid avoidable overtime."


Q4. How do you stop unsafe assignments?

Answer:
"Unsafe options are removed before scoring. The engine filters out license mismatches, people already scheduled, people on PTO, unit exclusions, rest-window violations, and cases that would drop the source unit below its coverage floor."


Q5. Does it consider certifications and licenses?

Answer:
"Yes. That is one of the first hard constraints. The engine only recommends people from the correct qualified bucket."


Q6. Does it consider unit experience?

Answer:
"Yes, through home unit, unit typology, cross-training, and seniority. The current demo does not yet model resident-level acuity, but it does capture the major operational experience signals that drive staffing fit."


Q7. Why would a Long-Term person not be ideal for Subacute?

Answer:
"Because Subacute is higher-acuity and more specialized. In the current scoring logic, Long-Term-only to Subacute is treated as a severe clinical mismatch, while Subacute to Long-Term is considered more acceptable."


Q8. What if there are not enough people?

Answer:
"Then the system produces the best possible schedule from the available pool and surfaces the remaining gaps clearly. It does not hide staffing shortages with unsafe assignments."


Q9. Does it remove human judgment?

Answer:
"No. It is explicitly human-in-the-loop. The system recommends and explains, but the coordinator makes the final decision."


Q10. Why require a reason for selecting someone other than the top-ranked candidate?

Answer:
"Because that preserves accountability and helps track when human operational context outweighs the ranking. It also creates a record for future refinement."


Q11. Does the system learn from overrides?

Answer:
"The current demo logs overrides and decisions, which creates the foundation for future learning and refinement. Right now the willingness signal is intentionally low-weight and neutral until enough history is available."


Q12. Can the ranking priorities be changed?

Answer:
"Yes. The scoring weights are configurable. If Parker decides that a certain factor should matter more or less, those weights can be adjusted."


Q13. Does proximity matter a lot?

Answer:
"It matters, but only lightly in the current configuration. It is mainly a tiebreaker. Parker's larger priorities are overtime control and clinical fit."


Q14. What data does the system need?

Answer:
"It uses data that facilities already maintain: staff roster, licenses, employment class, home unit, cross-training, schedule entries, hours ledger, call-out records, PTO, exclusions, and unit information."


Q15. What happens if the data in the database is outdated?

Answer:
"Then the recommendation quality will reflect that. The system depends on current staffing data, so good data hygiene and integration quality are important."


Q16. Does the monthly scheduler already handle every future leave scenario?

Answer:
"Not every future scenario. The current demo focuses on the core scheduling and replacement logic first. Deeper leave-impact planning is one of the logical next refinements."


Q17. Is the current UI the final product?

Answer:
"No. The current UI is enough to validate and demonstrate the workflow. The Marvel design represents a future design direction, but the purpose of this phase is to validate the algorithmic logic."


Q18. Why is this better than a scheduler doing it manually?

Answer:
"Because it reduces memory-based decision making, applies the same rules consistently every time, surfaces better options faster, and explains the reasoning instead of depending on whoever happens to be on shift."


Q19. Can this work with Parker's existing scheduling system?

Answer:
"That is the intended direction. The current demo is positioned as an intelligence layer that works alongside the existing scheduling workflow rather than replacing it outright."


Q20. What is the strongest value shown in this demo today?

Answer:
"The strongest immediate value is same-day call-out support with explainable ranked replacements. That is the clearest operational win in the current phase."
